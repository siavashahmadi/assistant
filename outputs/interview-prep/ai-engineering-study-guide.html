RAG Architecture
Retrieval-Augmented Generation — your chatbot's backbone
Your Bullet: "Collaborated with AI Engineers to build an agent-based LLM+RAG chatbot using LangChain and LlamaIndex, achieving 85% response accuracy validated through internal subject matter testing"

What is RAG and Why Does It Exist?

Think of an LLM like a brilliant student who memorized every textbook before exam day — but they graduated in 2023. They know nothing about your company's internal docs. RAG is like giving that student an open-book section: they can now look stuff up before answering.

RAG solves the fundamental problem of LLMs: they only know what they were trained on. If you need an AI to answer questions about your HVAC documentation, product manuals, or internal processes, the LLM alone can't do it. RAG lets you inject relevant context at query time without retraining the model.

The RAG Pipeline — Step by Step

📄 Documents → ✂️ Chunking → 🔢 Embedding → 📦 Vector Store → 🔍 Retrieval → 🤖 Generation

Step 1 — Ingestion & Chunking

You take your raw documents (PDFs, manuals, docs) and split them into smaller pieces. Why? Because LLMs have a limited context window (e.g., 4K-128K tokens). You can't feed the entire HVAC manual in every time. Plus, smaller chunks = more precise retrieval.

  • Chunk size matters: Too small (50 tokens) → loses context. Too large (2000 tokens) → dilutes relevance. Sweet spot is typically 256-512 tokens with overlap.
  • Overlap: You overlap chunks by ~50-100 tokens so you don't slice an important idea in half. Think of it like overlapping tiles on a roof.
  • Strategies: Fixed-size, sentence-based, semantic (split on topic boundaries), recursive character splitting (LangChain's default).

Step 2 — Embedding

Each chunk gets converted into a vector — a list of numbers (e.g., 1536 floats for OpenAI's text-embedding-ada-002) that represent the meaning of that text in high-dimensional space.

Imagine mapping every sentence onto a 3D globe. Sentences about "HVAC temperature" land near each other, while sentences about "billing" land on the other side. Except it's not 3 dimensions — it's 1536.
  • Models: OpenAI text-embedding-ada-002 / text-embedding-3-small, Cohere Embed, open-source like sentence-transformers
  • Key concept: Semantically similar text → vectors that are close together (measured by cosine similarity)

Step 3 — Vector Storage

Store those vectors in a specialized database optimized for similarity search.

  • Pinecone: Fully managed, serverless, scales automatically. You used this.
  • FAISS: Facebook's open-source library. Runs in-memory, blazing fast for local/smaller datasets. You also listed this.
  • Others: Chroma, Weaviate, Qdrant, Milvus, pgvector (PostgreSQL extension)

Index types in FAISS: Flat (exact, slow), IVF (approximate, clusters vectors), HNSW (graph-based, fast). Trade-off is always speed vs. accuracy.

Step 4 — Retrieval

When a user asks a question, their query gets embedded using the same model, then you search the vector store for the top-K most similar chunks.

  • Similarity metrics: Cosine similarity (most common), dot product, Euclidean distance
  • Top-K: Typically retrieve 3-5 chunks. More = more context but also more noise.
  • Re-ranking: Optional step — use a cross-encoder or re-ranking model to re-order retrieved chunks by actual relevance (not just embedding proximity).
  • Hybrid search: Combine vector similarity with keyword search (BM25) for better recall. Your chatbot may have used this.

Step 5 — Augmented Generation

Stuff those retrieved chunks into the LLM's prompt as context, then the LLM generates an answer grounded in your data.

  • Prompt structure: System message + "Here is relevant context: [chunks]" + User question
  • Grounding: The LLM should only answer based on the provided context, not hallucinate from training data
  • Context window management: If chunks are too big, you exceed the window. That's why chunk size matters upstream.

How to Talk About Your RAG Chatbot in the Interview

  • Architecture: "We built an agent-based RAG system — LangChain orchestrated the pipeline, LlamaIndex handled document ingestion and indexing, and we stored embeddings in a vector database for similarity-based retrieval."
  • Why agent-based: "An agent can decide when to retrieve vs. answer from memory, and can chain multiple tools — so it wasn't just Q&A, it could route to different knowledge bases or escalate."
  • Validation: "We validated against internal SMEs — had domain experts score responses for accuracy and relevance, hitting 85%."
  • What you'd improve: "I'd add evaluation frameworks like RAGAS for automated metrics (faithfulness, answer relevancy, context precision) and implement hybrid search for better recall."

LangChain & LangGraph
Orchestration frameworks for LLM applications

LangChain — The Framework

LangChain is like Express.js but for LLM apps. Express gives you routing, middleware, request/response patterns for web servers. LangChain gives you chains, prompts, memory, and tool-calling patterns for AI apps. You could build everything from scratch, but the framework saves you months.

Core Concepts You Should Know

  • LLMs / Chat Models: Wrappers around model APIs (OpenAI, Anthropic, etc.) with a unified interface. ChatOpenAI(model="gpt-4")
  • Prompt Templates: Reusable prompt structures with variables. Not just f-strings — they support chat messages, few-shot examples, and partial filling.
  • Chains: Composable sequences of steps. Legacy: LLMChain. Modern: LCEL (LangChain Expression Language) using the pipe operator: prompt | llm | output_parser
  • Tools & Agents: Tools are functions the LLM can call (search, calculator, database lookup). Agents decide which tool to call and when based on the user's input. This is what makes your chatbot "agent-based."
  • Memory: Conversation history management. Options: buffer (keep everything), summary (compress old messages), window (keep last N turns).
  • Retrievers: Interface to your vector store. Your RAG retriever fetches relevant documents and feeds them to the chain.
  • Output Parsers: Structure LLM output into JSON, lists, Pydantic models, etc.

LCEL — The Modern Way (Post-2023)

LangChain migrated from the old Chain classes to LCEL (LangChain Expression Language). This is the pipe-based declarative syntax:

chain = prompt | llm | StrOutputParser()

Benefits: streaming support, async, batching, easier debugging. If asked about LangChain evolution, mention you're aware of this shift from legacy chains to LCEL.

LangGraph — Stateful Agent Workflows

If LangChain chains are like a conveyor belt (linear: A → B → C), LangGraph is like a flowchart with decision points and loops. The agent can go back to a previous step, branch based on conditions, or run parallel tasks.
  • What it is: A library (from LangChain team) for building stateful, multi-step agent applications as graphs.
  • Nodes: Each node is a function (e.g., "retrieve documents", "generate answer", "check quality").
  • Edges: Define flow between nodes. Can be conditional — "if quality < threshold, go back to retrieve."
  • State: A shared state object passed through the graph. Each node reads/writes to it. Think of it as a mutable context that flows through your pipeline.
  • Checkpointing: Can save state mid-execution for human-in-the-loop workflows or recovery from failures.
  • Why use it over plain LangChain: When your agent needs loops, conditional routing, parallel execution, or persistence. A simple RAG chain doesn't need it. A complex agent that retrieves, evaluates, re-retrieves, then generates — that's LangGraph territory.

When to Mention LangGraph

If asked about more complex agent architectures or "how would you improve the chatbot," you can say: "For more complex multi-step reasoning, I'd consider LangGraph to build a graph-based workflow where the agent can loop through retrieval and quality-checking steps before returning a final answer."

LlamaIndex
The data framework for LLM apps

LlamaIndex vs. LangChain — Know the Difference

LangChain is like a Swiss Army knife — orchestration, agents, memory, tools, everything. LlamaIndex is like a specialized power drill — it does one thing exceptionally well: connecting your data to LLMs. Ingestion, indexing, querying. That's its superpower.
  • LlamaIndex excels at: Document loading (100+ connectors), smart chunking, multiple index types, query engines, response synthesis
  • LangChain excels at: Agent orchestration, tool calling, chain composition, memory management
  • In your chatbot: You likely used LlamaIndex for the ingestion/retrieval side and LangChain for the agent/orchestration side. This is a very common pattern.

Key LlamaIndex Concepts

  • Documents & Nodes: Documents are raw files. Nodes are the chunks after splitting. Each node has metadata (source, page number, etc.).
  • Indexes: The structure that organizes your nodes for retrieval.
    • VectorStoreIndex — most common, stores embeddings
    • SummaryIndex — summarizes all docs, good for "tell me about everything"
    • TreeIndex — hierarchical, good for documents with structure
    • KeywordTableIndex — keyword-based retrieval
  • Query Engine: Takes user query → retrieves relevant nodes → synthesizes response. This is the core retrieval+generation loop.
  • Response Synthesizer: How retrieved chunks become a final answer. Options: "compact" (stuff into one prompt), "refine" (iterate through chunks), "tree_summarize" (hierarchical).

Vector Databases
FAISS & Pinecone — where embeddings live

Why Vector Databases Exist

A regular database is like a filing cabinet — great for exact lookups ("find invoice #1234"). A vector database is like a librarian who understands meaning — "find me documents similar to this concept." It works on proximity, not exact match.

FAISS (Facebook AI Similarity Search)

  • What: Open-source library (not a managed service) for efficient similarity search
  • Runs: In-memory, on your machine. No server needed.
  • Best for: Prototyping, smaller datasets, low-latency requirements, when you don't need cloud infrastructure
  • Index types: IndexFlatL2 (exact, brute force), IndexIVFFlat (approximate, uses clusters), IndexHNSW (graph-based, fast)
  • Limitation: No built-in persistence (you serialize to disk), no metadata filtering out of the box, no managed scaling

Pinecone

  • What: Fully managed vector database as a service
  • Best for: Production deployments, when you need scaling, metadata filtering, and don't want to manage infrastructure
  • Features: Serverless option, namespaces, metadata filtering (e.g., "find similar docs but only from building X"), hybrid search
  • API: index.upsert(vectors), index.query(vector, top_k=5, filter={"building_id": "123"})

When Asked "Why Pinecone vs. FAISS?"

"FAISS is great for prototyping and in-memory speed, but for production with metadata filtering requirements — like filtering by building, sensor type, or document category — Pinecone's managed infrastructure and filtering capabilities made more sense. We didn't want to manage our own vector index infrastructure on top of everything else."

Generative AI Data Labeling
Your OpenAI-powered sensor normalization pipeline
Your Bullet: "Deployed a generative-AI labeling workflow using OpenAI API to label unstructured IoT data for analysis, increasing speed by 67% and automating a manual process while maintaining 90% labeling accuracy."

The Problem & Your Solution

Imagine 200 buildings, each with sensors from different manufacturers. One calls temperature "TMP", another says "TEMP", another says "ZoneTemp_1". A human had to manually map each of these to a standard label. You used an LLM to automate this classification task.

How It Worked

  • Input: Raw, unstructured sensor tag names from IoT devices across buildings
  • Process: Feed sensor tags + context documents (what each building's naming convention looks like) to OpenAI API
  • Pattern: This is essentially a few-shot classification task — you give the LLM examples of correct mappings (context/RAG) and ask it to classify new, unseen tags
  • Output: Standardized labels mapped to each raw sensor tag
  • Validation: Human review loop — SMEs checked a sample for accuracy, achieving 90%

Technical Details to Be Ready For

  • "How did you handle edge cases?" → Some tags were ambiguous (e.g., "SF" could be supply fan or square footage). We provided building-specific context docs so the LLM had manufacturer info to disambiguate.
  • "Why not a traditional ML classifier?" → The tag vocabulary was constantly changing as new buildings onboarded. An LLM with RAG context adapts without retraining. A traditional classifier would need new training data each time.
  • "How did you measure accuracy?" → Sampled outputs, had domain experts validate. The 90% accuracy was measured against human expert labels on a holdout set.
  • "What about cost?" → OpenAI API calls have cost. We batched requests and used the most cost-effective model tier for this classification task since it didn't need GPT-4-level reasoning.

IoT Data Pipeline
RabbitMQ, VOLTTRON, 10M+ data points/day
Your Bullet: "Designed scalable, RESTful APIs using Python and Flask for large-scale HVAC data streamed through RabbitMQ to a PostgreSQL database for fault detection and diagnosis that delivered 25% cost savings."

The Architecture

Think of your pipeline like a postal system. Sensors are the senders (thousands of letters/data points). RabbitMQ is the post office — it receives everything, sorts it, and routes it to the right department. VOLTTRON is the backup mailbox — if the post office goes down, it queues the mail until it's back. Your Flask services are the departments that read, process, and file the mail.
🏢 Building Sensors → 📡 VOLTTRON Agent → 🐰 RabbitMQ → 🔄 Normalization → ⚠️ Fault Detection → 🗄️ PostgreSQL / S3

RabbitMQ — Message Broker

  • What: Open-source message broker that implements AMQP (Advanced Message Queuing Protocol)
  • Why use it: Decouples producers (sensors) from consumers (your services). If your processing service goes down, messages queue up instead of being lost.
  • Key concepts: Exchanges (routing rules), Queues (where messages wait), Bindings (rules connecting exchanges to queues), Consumers (your services pulling messages)
  • Exchange types: Direct (exact routing key match), Topic (pattern matching), Fanout (broadcast to all queues), Headers
  • In your system: Sensors publish data → RabbitMQ routes to appropriate processing queues → consumers pick up and process

VOLTTRON

  • What: Open-source platform (DOE-funded) for distributed sensing and control in buildings
  • Your usage: Deployed at the edge (on-site) for message replay — when Wi-Fi drops, VOLTTRON queues HTTP requests locally and replays them when connectivity returns
  • Why it matters: Solves the #1 IoT problem — unreliable networks. Without this, you lose data during outages.
  • Key concept: Agent-based architecture — each VOLTTRON "agent" is an independent service that can be deployed, updated, and monitored separately

Fault Detection & Diagnosis

  • What: Algorithms that analyze HVAC sensor data to detect when equipment is malfunctioning
  • Examples: Simultaneous heating and cooling (waste), stuck dampers, sensor drift
  • Impact: The 25% cost savings — detecting and alerting on these faults before they waste energy

AWS Architecture
Glue, Athena, S3, hot/cold storage strategy
Your Bullet: "Migrated data to S3 using AWS Glue to save 57% on cost, queried via Athena-backed APIs for analytics."

Hot/Cold Storage Strategy

Think of it like your closet. Clothes you wear this week (hot data) stay in your dresser (RDS Aurora) — fast access, premium space. Clothes from last season (cold data) go in storage bins in the garage (S3) — cheap, still accessible, just takes a bit longer to dig through.

The Architecture

  • Hot path (RDS Aurora): Last 7 days of sensor data. Sub-second queries. This is what the real-time dashboards hit.
  • Cold path (S3): Historical data beyond 7 days. Cheap storage (~$0.023/GB/month vs. Aurora's much higher cost). This is where the 57% savings came from.
  • Migration: AWS Glue ETL jobs run on a schedule, moving aged-out data from Aurora → S3 in Parquet format.
  • Querying cold data: Athena queries S3 directly using SQL. No server to manage — pay per query. Partitioned by date for efficiency.

AWS Glue

  • What: Serverless ETL (Extract, Transform, Load) service
  • In your system: Scheduled jobs that extract data from RDS Aurora, transform it into Parquet (columnar, compressed format), and load it into S3
  • Crawlers: Glue Crawlers discover schema automatically and populate the Glue Data Catalog
  • Why Parquet: Columnar format → only reads the columns you query (not entire rows). Huge performance gain for analytics queries.

AWS Athena

  • What: Serverless query engine that runs SQL directly on S3
  • Key feature: No infrastructure to manage. You literally write SQL and it scans S3.
  • Pricing: $5 per TB scanned — so partitioning and Parquet format are critical to keep costs down
  • In your system: Analytics APIs call Athena for historical trend data, cost analysis, long-term performance reports

Other AWS Services in Your Stack

  • EC2: Compute instances running your Flask backend
  • S3: Object storage for cold data and static assets
  • Lambda: Serverless functions for event-driven tasks (e.g., triggering on new data arrival)
  • CloudFront: CDN for your React frontend — caches static assets at edge locations
  • RDS Aurora: Managed PostgreSQL with auto-scaling replicas
  • Redis: In-memory caching for frequently accessed data (dashboard configs, user sessions)

Full-Stack Architecture
React, Flask, APIs, Three.js

Your Full-Stack at Joulea

Frontend: React + TypeScript + Redux

  • React: Component-based UI library. You built reusable components with Material UI design system.
  • Redux: Centralized state management. Actions → Reducers → Store → Components. Used for complex cross-component state (building data, user auth, sensor readings).
  • TypeScript: Static typing on top of JavaScript. Catches bugs at compile time, self-documenting code, better IDE support.
  • Three.js: 3D rendering library. You used it for building visualizations — rendering 3D models of commercial buildings with sensor data overlays.
  • Performance: Implemented memoization (React.memo, useMemo, useCallback) to prevent unnecessary re-renders, especially important with real-time data updates.

Backend: Python Flask

  • Flask: Lightweight Python web framework. "Micro" framework — you add what you need (SQLAlchemy, Flask-CORS, etc.)
  • SQLAlchemy ORM: Maps Python classes to database tables. Write Python, it generates SQL.
  • RESTful API design: Resources as URLs (/api/buildings/{id}/sensors), HTTP methods (GET, POST, PUT, DELETE), status codes (200, 201, 400, 404, 500)
  • API patterns: Authentication middleware, request validation, error handling, pagination for large datasets, CORS configuration

T-Mobile Full Stack

  • Angular: Full-featured framework (vs React's library). Two-way data binding, dependency injection, TypeScript by default, opinionated project structure.
  • Spring Boot: Java framework for microservices. Auto-configuration, embedded server, production-ready features (metrics, health checks).
  • Apigee: API gateway/management platform. Rate limiting, authentication, analytics, API versioning at the gateway level.
  • OAuth 2.0: Authorization framework. User authenticates → gets access token → includes token in API calls → backend validates. Flows: Authorization Code (web), Client Credentials (service-to-service).
