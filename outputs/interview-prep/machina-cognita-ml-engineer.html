RAG Architecture — Deep Dive

You built this. Be ready to whiteboard every component.

Why RAG instead of fine-tuning? Fine-tuning bakes knowledge into model weights — expensive, slow to update, risks catastrophic forgetting. RAG retrieves fresh context at query time. For domain-specific, frequently-updated knowledge (like military doctrine or IoT building data), RAG wins.

1. Chunking: Documents split into overlapping chunks (~200-500 tokens). Chunk size tradeoffs — too small loses context, too large dilutes relevance. Overlap (e.g., 50 tokens) prevents context loss at boundaries.

2. Embeddings: Each chunk encoded into a dense vector (e.g., OpenAI text-embedding-3-small, 1536 dims). Similar semantic meaning = close vectors in high-dimensional space. Model choice matters — domain-specific embeddings outperform generic ones.

3. Vector Store: FAISS (what you used) — Facebook's library for approximate nearest neighbor (ANN) search. Flat index for exact search, IVF/HNSW for scalable approximate. Also: Pinecone, Weaviate, Chroma, pgvector.

4. Retrieval: Query embedded, top-k nearest chunks returned. Can add re-ranking (cross-encoder scores retrieved chunks for relevance) and MMR (maximal marginal relevance) to reduce redundancy.

5. Generation: Retrieved chunks + original query assembled into prompt. Sent to LLM. Prompt engineering matters here — instructing the LLM to ground answers in the provided context, not hallucinate.

6. Evaluation: RAGAS framework — Faithfulness (does answer match retrieved docs?), Answer Relevancy (does it answer the question?), Context Precision/Recall. You used SME validation for 85% accuracy — frame this as a human-in-the-loop eval approach.

Your story: LangChain orchestration, LlamaIndex for document ingestion + indexing, FAISS vector store, agent-based routing (different agents for HVAC queries vs. analytics vs. support). 85% accuracy via SME validation. Mention the chunking and embedding strategy decisions you made.

LLM Fundamentals — Transformers, Attention, Fine-tuning

Know the architecture well enough to explain tradeoffs.

Transformer architecture: Input tokens → embeddings → positional encoding → N layers of (multi-head self-attention + feed-forward network) → output. Attention lets each token "look at" all other tokens to determine relevance — this is what makes LLMs context-aware.

Self-attention: For each token, compute Query (Q), Key (K), Value (V) projections. Attention score = softmax(QK^T / sqrt(d_k)) * V. Scaled dot-product prevents vanishing gradients. Multi-head = run N attention heads in parallel, then concatenate.

BERT vs GPT: BERT = encoder-only, bidirectional (reads full context). Good for classification, NER, embeddings. GPT = decoder-only, autoregressive (left-to-right). Good for generation. MCT's SMARTS uses BERT for NLP classification tasks.

Fine-tuning approaches:
— Full fine-tuning: Update all weights. Expensive, risky (catastrophic forgetting), needs lots of data.
— LoRA (Low-Rank Adaptation): Freeze base model, inject small trainable matrices into attention layers. Dramatically reduces trainable parameters (e.g., 0.1% of original). Fast, cheap, reversible.
— QLoRA: LoRA + quantize the base model to 4-bit. Enables fine-tuning large models (70B+) on a single GPU. Standard approach for resource-constrained environments.
— RLHF (Reinforcement Learning from Human Feedback): Train a reward model from human preferences, then use PPO to fine-tune the LLM to maximize reward. How ChatGPT was aligned. Expensive but powerful for behavioral alignment.

Your connection: You've used fine-tuned models via API (OpenAI) and built RAG as an alternative to fine-tuning. Frame your choice of RAG over fine-tuning as a deliberate architectural decision based on data freshness, cost, and update frequency constraints.

Reinforcement Learning — Concepts You Need for MCT

STAMPEDE + SMARTS use RL. Know the vocabulary, be honest about experience.

Core loop: Agent observes state (s) → selects action (a) → environment returns reward (r) + next state (s'). Goal: learn a policy π(a|s) that maximizes cumulative reward over time.

Key concepts:
— Q-learning: Model-free. Learn Q(s,a) = expected total reward from state s taking action a. Update via Bellman equation: Q(s,a) ← r + γ·max Q(s',a'). DQN = Q-learning with neural network approximator.
— Policy Gradient (REINFORCE, PPO, A3C): Directly learn the policy π. PPO (Proximal Policy Optimization) is the industry workhorse — stable, sample-efficient, used in RLHF.
— Model-free vs. model-based: Model-free learns directly from experience (slower, less data-efficient). Model-based learns a world model first, then plans within it (more data-efficient, harder to train).
— Exploration vs. exploitation: ε-greedy (random action with prob ε), UCB, Thompson sampling.
— Reward shaping: Designing the reward function is often the hardest part. Sparse rewards (only at goal) = hard to learn. Dense rewards = easier but can be gamed.

MCT context: STAMPEDE uses RL for motion prediction — an agent learning movement patterns. SMARTS uses RL + NLP for tactical decision support — the RL component likely rewards strategies that align with doctrine.

Your honest response: "I understand the RL framework conceptually and I've studied the literature, but I haven't built RL systems in production. What I bring is the production engineering infrastructure — reliable data pipelines, model serving, evaluation frameworks — that makes RL research deployable. I'd ramp the RL implementation side fast."

NLP Pipeline Concepts

SMARTS, ALICE, Marine-EDGE are all NLP projects.

Pipeline stages: Raw text → Tokenization → Encoding → Task-specific head → Output

Tokenization: BPE (Byte-Pair Encoding) used by GPT — merges frequent character pairs. WordPiece used by BERT — splits unknown words into subwords. SentencePiece is language-agnostic. Vocab size tradeoffs: larger vocab = more precise tokens but bigger embedding matrix.

NER (Named Entity Recognition): Tag each token with an entity type. "USS Enterprise deployed near Okinawa" → [USS Enterprise: SHIP], [Okinawa: LOCATION]. MCT's ALICE almost certainly uses NER to extract entities from intel reports. Approaches: BERT + token classification head, spaCy, CRF layers.

Intent Classification: Classify whole sentence intent. "What is the threat level at grid 4-7?" → QUERY_THREAT_LEVEL. Used in SMARTS for routing tactical queries. BERT + CLS token classification.

Relation Extraction: Beyond NER — extract relationships between entities. "Ship A is-15nm-from Base B" → (Ship A, DISTANCE, Base B). Used in knowledge graphs (NOVA, Marine-EDGE).

BERT architecture detail: 12 transformer encoder layers (BERT-base). Input = [CLS] + tokens + [SEP]. Pre-trained on Masked Language Modeling (predict masked tokens) + Next Sentence Prediction. Fine-tune by adding task head on top of [CLS] or per-token representations.

Your connection: Your GenAI labeling pipeline is essentially an NLP classification system — feeding unstructured IoT sensor descriptions to an LLM and extracting structured labels. Same pattern as ALICE extracting structured intel from raw text.

Explainable AI (XAI)

SCORED is literally an XAI project. This will come up.

Why it matters in defense: A black box recommendation to engage a target is legally and ethically unacceptable. Commanders need to understand why the model said what it said. XAI is not optional — it's a hard requirement for DoD AI adoption.

SHAP (SHapley Additive exPlanations): Based on game theory. Assigns each feature a Shapley value = its contribution to the prediction. Model-agnostic. Computationally expensive but theoretically sound. "Feature X pushed the prediction 0.3 toward class A."

LIME (Local Interpretable Model-agnostic Explanations): Fit a simple interpretable model (linear) locally around the prediction. Faster than SHAP, less globally consistent. Good for quick per-prediction explanations.

Attention Visualization: For transformers — visualize which tokens the model attended to. Intuitive but contested: high attention ≠ high importance necessarily. BertViz is a common tool.

Saliency Maps (for vision/signals): Highlight which input regions most influenced the output. For SCORED (SIGINT/RF signal DL) — show which parts of the RF spectrogram drove the classification. Gradient-based (vanilla gradients, integrated gradients) or perturbation-based (occlusion).

Counterfactuals: "What would have to change for the model to predict differently?" Useful for debugging and trust-building.

Your connection: At Joulea you built outputs that non-technical stakeholders (facilities managers, energy teams) had to act on. You understand the human-trust requirement. Mention that you designed outputs with explainability in mind — e.g., the chatbot answers cited source documents, not just generated text.

Vector Databases and Similarity Search

Core infrastructure for every LLM application at MCT.

Why vectors? Semantic similarity can't be found with SQL LIKE queries. You need to find "closest meaning" not "exact match." Vector DBs store high-dimensional embeddings and support fast similarity search.

Cosine similarity: Dot product of normalized vectors. Range [-1, 1]. Measures angle between vectors, not magnitude — so a short and long document with similar meaning score the same. Most common metric for text embeddings.

Euclidean distance (L2): Geometric distance. Better for image embeddings. Less common for text.

ANN algorithms:
— FAISS (what you used): Facebook AI. Flat index = exact brute-force. IVF (Inverted File Index) = cluster vectors, search only nearby clusters. HNSW (Hierarchical Navigable Small World) = graph-based, fast queries, good recall.
— HNSW: Builds a multi-layer graph where each node connects to its nearest neighbors. Search starts at top layer (coarse), drills down. Sub-linear query time.
— Tradeoff: Exact search = perfect recall, O(n) time. ANN = slight recall loss, O(log n) time.

Managed vector DBs: Pinecone (cloud-native), Weaviate (hybrid search), Chroma (local-first), Qdrant, pgvector (Postgres extension). FAISS is a library, not a DB — you need to manage persistence yourself.

Your story: Used FAISS for the RAG chatbot. Know why you chose it (local deployment, no external API calls, full control) and what tradeoffs it created (you manage persistence, no built-in metadata filtering).

Model Evaluation — The Right Metrics

Be ready to justify your 85% accuracy claim.

Classification metrics:

Precision: Of all positive predictions, how many were correct? TP / (TP + FP). Optimize when false positives are costly (e.g., false threat alerts).

Recall: Of all actual positives, how many did we catch? TP / (TP + FN). Optimize when false negatives are costly (e.g., missing a real threat).

F1: Harmonic mean of precision and recall. Balanced metric when both matter.

For RAG specifically: Standard classification metrics don't apply. Use RAGAS (Faithfulness, Answer Relevancy, Context Precision, Context Recall) or custom human evaluation. You used SME validation — say this is a deliberate choice for domain-specific accuracy that automated metrics can't capture reliably.

For generation tasks: BLEU/ROUGE (n-gram overlap) — often inadequate. LLM-as-judge (use another LLM to evaluate quality) is emerging as standard.

Confusion matrix: Know how to read it. Important for multi-class classification in NLP/NER tasks.

Your story: 85% accuracy at Joulea — be ready to say what metric this was, how it was measured, and what the 15% failure modes looked like. "We defined accuracy as: did the answer correctly address the user's question as judged by a domain SME? We tested on 200 held-out queries across all intent categories."

ML System Design

Technical interviews often end with "design an LLM pipeline for X."

Standard LLM pipeline components:

Ingestion: Document parsing (PyMuPDF, Unstructured.io) → chunking → embedding → vector store insert. Incremental updates vs. full re-index.

Inference: Query → embed → retrieve top-k → re-rank → construct prompt → LLM → post-process output → return.

Serving: FastAPI or Flask wrapper, async for throughput, caching for repeated queries, rate limiting for cost control.

Observability: Log every query + retrieved chunks + final response. Track latency, token usage, cost per query. Alert on quality degradation.

For MCT specifically: Air-gapped deployment? Need local models (Llama 3, Mistral) + local vector stores (FAISS, Chroma). Cannot call OpenAI from a SCIF.

What do you know about transformers and how attention works?
"At the core, attention is how the model figures out which parts of the input are relevant to each other. For each token, you compute three vectors — Query, Key, and Value. The attention score between two tokens is the dot product of their Q and K vectors, scaled by the square root of the dimension to prevent vanishing gradients, then softmaxed to get a probability distribution. You use that distribution to take a weighted sum of the Value vectors — so each output token is informed by all input tokens, weighted by relevance.

Multi-head attention runs this process in parallel N times with different learned projections, then concatenates. This lets the model attend to different aspects simultaneously — syntax in one head, semantics in another.

For MCT's use case: BERT is encoder-only — it reads the full sequence bidirectionally, which makes it excellent for classification and NER tasks like SMARTS. GPT-style models are decoder-only and autoregressive — better for generation tasks like ALICE's intel report generation."

Watch For

Clearance Blocker

If they need active Secret clearance to start — that's 6-18 months you can't work on classified projects. If there's no unclassified work in the meantime, this is a non-starter.

Salary Below Your Floor

SBIR companies pay less. If their range is $80-$110K, that's well below your $130K floor. BUT — if the ML work is genuinely strong and clearance path is real, it could be worth it for the resume and technical growth.

PhD Research Culture

If every question is about published papers, novel research, or thesis work — this role may want a researcher, not an engineer. Your strength is production ML. If the vibe is "what papers have you read," this may not be your fit.

Location Rigidity

Carlsbad is San Diego metro. If they require 5 days/week on-site with no flexibility, factor in the full cost of a CA move before committing. Ask about hybrid early.
