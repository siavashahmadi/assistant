The ML Pipeline: End to End

ANALOGY

Think of an ML pipeline like building a restaurant. You don't just cook food (train a model). You first source ingredients (collect data), prep them (clean and transform), taste-test the recipe (evaluate), and then serve to customers (deploy). If any step is sloppy, the final dish suffers. Most ML engineering is the prep work, not the cooking.

The standard ML pipeline has these stages. Every interviewer expects you to know this flow:

Raw Data
→
Clean & Transform
→
Feature Engineering
→
Train / Validate
→
Evaluate
→
Deploy / Monitor

Stage by Stage

1. Data Collection - You gather your raw inputs. At Joulea, this was IoT sensor data streaming through RabbitMQ: temperature readings, humidity, energy consumption, pressure values, all coming in from 200+ buildings at 10M+ data points per day.

2. Data Cleaning & Transformation - Raw data is always messy. Sensors drop out, send duplicates, use different naming conventions (TMP vs TEMP vs Temperature). You normalize formats, handle missing values (imputation or removal), remove outliers, and align timestamps. This is where your generative AI labeling workflow fits in: using the OpenAI API to standardize vendor-specific naming.

3. Feature Engineering - This is where you go from raw readings to meaningful signals. More on this in the Feature Engineering section.

4. Model Training & Validation - Split your data, pick an algorithm, train it, tune hyperparameters. More detail in the Train/Test and Scikit-learn sections.

5. Evaluation - Measure how good the model is using metrics like precision, recall, F1 score, accuracy. Covered in the Metrics section.

6. Deployment & Monitoring - Put the model into production. At Joulea, this means the model runs against incoming sensor data in real-time or near-real-time, flagging HVAC faults. You monitor for model drift (when the model's accuracy degrades over time because the incoming data distribution changes).

INTERVIEW TIP

When asked "walk me through an ML pipeline," don't jump straight to the model. Start with the data. Interviewers want to hear that you understand the full lifecycle, not just the scikit-learn `.fit()` call. Your Joulea experience is perfect here because you literally built the data ingestion pipeline that feeds the ML system.

Train/Test Splits

ANALOGY

Imagine studying for an exam. If you practice using the exact same questions that will appear on the test, you'll ace it, but you haven't actually learned anything. You've just memorized answers. A train/test split is like studying with one set of problems (training set) and then proving you learned the concepts by solving different problems you've never seen (test set).

The Core Idea

You split your dataset into portions so the model is evaluated on data it has never seen during training. This tells you whether the model has actually learned patterns vs. just memorizing your data (overfitting).

Common Splits

SplitRatioPurpose
Train 70-80% Model learns patterns from this data
Validation 10-15% Tune hyperparameters, compare model variants. Like a practice exam.
Test 10-15% Final evaluation only. Touch this ONCE. This is the real exam.

In Code (Scikit-learn)

from sklearn.model_selection import train_test_split

# Simple 80/20 split
X_train, X_test, y_train, y_test = train_test_split(
    features, labels,
    test_size=0.2,
    random_state=42  # reproducibility
)

# With validation set: first split 80/20, then split train into 75/25
X_train, X_test, y_train, y_test = train_test_split(
    features, labels, test_size=0.2, random_state=42
)
X_train, X_val, y_train, y_val = train_test_split(
    X_train, y_train, test_size=0.25, random_state=42
)
# Result: 60% train, 20% validation, 20% test

Cross-Validation

When you don't have a ton of data, k-fold cross-validation is better than a single split. You divide data into k folds (usually 5 or 10), train on k-1 folds, validate on the remaining fold, and repeat k times. Every data point gets used for both training and validation.

from sklearn.model_selection import cross_val_score

scores = cross_val_score(model, X, y, cv=5, scoring='f1')
print(f"Mean F1: {scores.mean():.3f} (+/- {scores.std():.3f})")
INTERVIEW TIP

For time-series data like HVAC readings, you cannot do random splits. You must split chronologically: train on older data, test on newer data. Otherwise you're "leaking" future information into training. This is called temporal leakage and it's a classic mistake. Mention this proactively and you'll stand out.

YOUR JOULEA CONTEXT

"We had millions of data points per day from 200+ buildings. When building fault detection models, we split chronologically, training on historical data and validating on recent unseen data, because random splits would leak future sensor patterns into the model. We used time-based cross-validation to account for seasonal variation in HVAC behavior."

Precision, Recall, F1 Score

ANALOGY

Imagine you're a smoke detector. Precision = "When I beep, is there actually a fire?" (Are my alarms real or false alarms?). Recall = "When there IS a fire, do I beep?" (Am I catching all the real fires?). A smoke detector with high precision but low recall barely ever beeps, but misses real fires. One with high recall but low precision catches every fire but also beeps when you make toast. You need a balance.

The Confusion Matrix

Everything flows from four numbers:

Predicted PositivePredicted Negative
Actually Positive True Positive (TP) False Negative (FN) - Missed it
Actually Negative False Positive (FP) - False alarm True Negative (TN)

The Metrics

Precision = TP / (TP + FP)
"Of everything I flagged as positive, how many were actually positive?"
High precision = few false alarms.

Recall (also called Sensitivity) = TP / (TP + FN)
"Of all actual positives, how many did I catch?"
High recall = few missed cases.

F1 Score = 2 * (Precision * Recall) / (Precision + Recall)
The harmonic mean of precision and recall. It balances both. If either precision or recall is very low, F1 will be low too.

Accuracy = (TP + TN) / Total
Sounds great but is misleading with imbalanced data. If 95% of your HVAC readings are "normal" and 5% are "fault," a model that always predicts "normal" has 95% accuracy but catches zero faults. Useless.

Which to Optimize For?

DEPENDS ON THE COST OF ERRORS

Optimize Precision when false positives are expensive. Example: sending a maintenance crew to a building costs money. You don't want to dispatch them for a false alarm.

Optimize Recall when false negatives are dangerous. Example: missing a real HVAC fault could cause equipment failure or tenant complaints. You'd rather have some false alarms than miss a real problem.

Optimize F1 when you need a balance, which is the most common choice for fault detection.

In Code

from sklearn.metrics import (
    precision_score, recall_score, f1_score,
    classification_report, confusion_matrix
)

# Individual metrics
precision = precision_score(y_test, y_pred)
recall = recall_score(y_test, y_pred)
f1 = f1_score(y_test, y_pred)

# The full picture in one call
print(classification_report(y_test, y_pred))

# Output looks like:
#               precision    recall  f1-score   support
#     normal       0.97      0.95      0.96      1200
#      fault       0.72      0.81      0.76       150
#   accuracy                           0.93      1350
YOUR JOULEA CONTEXT

"For HVAC fault detection, we prioritized recall slightly over precision because a missed fault could lead to equipment degradation or wasted energy, and we'd rather flag something for a technician to review than let a real fault go unnoticed. We tracked F1 as our primary metric since we still needed reasonable precision to avoid alert fatigue for building managers."

Feature Engineering

ANALOGY

Think of raw sensor data like raw ingredients in a kitchen. A pile of flour, eggs, and sugar is useless to a restaurant customer. Feature engineering is the chef's work: combining, transforming, and presenting those raw ingredients into something meaningful. A "rolling average temperature over 24 hours" is a dish; a single temperature reading at 2:47 PM is just an egg.

What Is It?

Feature engineering is the process of transforming raw data into features (input variables) that make machine learning models more effective. It's widely considered the highest-leverage activity in applied ML. Better features beat better algorithms almost every time.

Feature Types for HVAC / Time-Series Data

Statistical Aggregations

Instead of individual readings, you compute summaries over time windows:

  • Rolling mean - Average temperature over the last 1hr, 6hr, 24hr
  • Rolling std deviation - How much is the reading fluctuating? High variance might signal a malfunctioning sensor.
  • Min / Max in window - Did the temperature spike or drop?
  • Rate of change (delta) - How fast is the temperature changing? A rapid drop could signal a fault.

Temporal Features

Time itself carries information:

  • Hour of day - HVAC behavior differs at 3 AM vs 3 PM
  • Day of week - Weekday vs weekend occupancy patterns
  • Season / month - Summer cooling vs winter heating
  • Is business hours? - Binary flag: 1 if 8AM-6PM weekday, 0 otherwise

Domain-Specific Features

  • Delta-T (supply vs return temperature) - A core HVAC metric. If the difference between supply and return air is abnormal, something's wrong.
  • Outdoor temp vs indoor temp ratio - Is the building responding appropriately to outside conditions?
  • Deviation from setpoint - How far is the actual reading from the target?
  • Equipment runtime hours - Longer runtimes can predict maintenance needs

Lag Features

Past values as input features. "What was the temperature 1 hour ago? 6 hours ago? Yesterday at the same time?" This helps the model learn temporal patterns.

In Code

import pandas as pd

# Rolling statistics
df['temp_rolling_mean_24h'] = df['temperature'].rolling(window=24).mean()
df['temp_rolling_std_24h']  = df['temperature'].rolling(window=24).std()
df['temp_rate_of_change']   = df['temperature'].diff()

# Temporal features
df['hour']         = df['timestamp'].dt.hour
df['day_of_week']  = df['timestamp'].dt.dayofweek
df['is_weekend']   = df['day_of_week'].isin([5, 6]).astype(int)
df['is_business_hours'] = (
    (df['hour'] >= 8) & (df['hour'] <= 18) & (~df['is_weekend'])
).astype(int)

# Domain features
df['delta_t'] = df['supply_temp'] - df['return_temp']
df['setpoint_deviation'] = df['actual_temp'] - df['setpoint_temp']

# Lag features
df['temp_lag_1h']  = df['temperature'].shift(1)
df['temp_lag_24h'] = df['temperature'].shift(24)
INTERVIEW TIP

If asked "what features did you engineer?", don't just say "rolling averages." Be specific: "We computed rolling 24-hour mean and standard deviation for temperature readings, created delta-T features comparing supply and return air temps, added temporal signals like hour-of-day and is-business-hours flags, and used lag features to capture how current readings compare to the same time yesterday. The temporal and delta-T features had the highest feature importance in our Random Forest model."

Scikit-learn in Practice

ANALOGY

Scikit-learn is to machine learning what Express.js is to backend APIs. It's the standard library that gives you a consistent interface for doing the most common ML tasks. Every model follows the same pattern: fit() to train, predict() to infer, score() to evaluate. Learn one, you know the API for all of them.

The Universal Pattern

# EVERY scikit-learn model follows this exact pattern:

# 1. Import the model class
from sklearn.ensemble import RandomForestClassifier

# 2. Create an instance with hyperparameters
model = RandomForestClassifier(
    n_estimators=100,    # number of trees
    max_depth=10,         # limit tree depth to prevent overfitting
    random_state=42
)

# 3. Train (fit) on training data
model.fit(X_train, y_train)

# 4. Predict on new data
y_pred = model.predict(X_test)

# 5. Evaluate
score = model.score(X_test, y_test)  # accuracy by default

Models You Should Know

ModelUse WhenProsCons
Random Forest Good default for classification and regression. Great first model to try. Handles non-linear data, feature importance built in, hard to overfit Slower to train on huge datasets, less interpretable than single tree
Gradient Boosting (XGBoost, LightGBM) When you need maximum performance on tabular data Usually highest accuracy for tabular, handles missing data Easier to overfit, more hyperparameters to tune
Logistic Regression Simple binary classification, baseline model, interpretability needed Fast, interpretable, good baseline Can't capture non-linear relationships without feature engineering
SVM (Support Vector Machine) Smaller datasets, high-dimensional features Works well in high dimensions Slow on large datasets, doesn't scale well

Preprocessing Pipeline

Real ML code uses scikit-learn Pipelines to chain preprocessing and modeling together. This prevents data leakage and makes your code reproducible:

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import RandomForestClassifier

pipeline = Pipeline([
    ('scaler', StandardScaler()),        # normalize features
    ('classifier', RandomForestClassifier(
        n_estimators=100, max_depth=10
    ))
])

# Fit entire pipeline (scaler + model) on training data
pipeline.fit(X_train, y_train)

# Predict - scaler automatically transforms before predicting
y_pred = pipeline.predict(X_test)

Hyperparameter Tuning

from sklearn.model_selection import GridSearchCV

param_grid = {
    'n_estimators': [50, 100, 200],
    'max_depth': [5, 10, 20, None],
    'min_samples_split': [2, 5, 10]
}

grid_search = GridSearchCV(
    RandomForestClassifier(), param_grid,
    cv=5, scoring='f1', n_jobs=-1
)
grid_search.fit(X_train, y_train)
print(f"Best params: {grid_search.best_params_}")
print(f"Best F1: {grid_search.best_score_:.3f}")

We used scikit-learn's Pipeline to chain StandardScaler with the classifier, and tuned hyperparameters using GridSearchCV with 5-fold time-series cross-validation, optimizing for F1 score."

Model Fine-Tuning

ANALOGY

Fine-tuning is like taking someone who speaks fluent English (a pre-trained model) and teaching them industry jargon for HVAC systems. You're not teaching them language from scratch. You're adding specialized knowledge on top of what they already know. This is WAY cheaper and faster than training from zero.

Two Contexts for "Fine-Tuning"

In interviews, "fine-tuning" can mean two different things depending on context:

1. Traditional ML Fine-Tuning (Scikit-learn world)

This means hyperparameter tuning: adjusting knobs on your model (learning rate, tree depth, number of estimators) to improve performance. This is what GridSearchCV does. It's the more common meaning in classical ML.

2. LLM Fine-Tuning (AI/NLP world)

This means taking a pre-trained foundation model (like GPT or an open-source model) and training it further on your own domain-specific data to specialize it. There are several approaches:

ApproachWhat It DoesCost
Full Fine-Tuning Update ALL model weights on your data Very expensive. Needs lots of data + GPUs.
LoRA / QLoRA Freeze most weights, train small adapter layers Much cheaper. Can fine-tune on a single GPU.
Prompt Tuning Learn optimal prompt embeddings, freeze model Cheapest. Model weights never change.
RAG (what you did) Don't change the model at all. Feed it relevant context at inference time. No training cost. Just retrieval infrastructure.
KEY DISTINCTION

RAG vs Fine-Tuning: RAG is "giving the model a reference sheet during the exam." Fine-tuning is "teaching the model before the exam." RAG is better when your knowledge base changes frequently (like building documentation). Fine-tuning is better when you need the model to learn a consistent style or behavior pattern.

YOUR JOULEA CONTEXT

"For our chatbot, we chose RAG over fine-tuning because our building documentation changed frequently, and we needed the system to always reference the latest specs without retraining. However, for the data labeling workflow, we effectively used prompt engineering as a form of lightweight tuning, providing detailed examples and context in the system prompt to guide the OpenAI API's classification behavior for our specific sensor naming conventions."

Classification Models

ANALOGY

Classification is like a doctor diagnosing a patient. The patient presents symptoms (features), and the doctor categorizes them into a diagnosis (class label). Binary classification: "Is this patient sick or healthy?" Multi-class: "Does this patient have a cold, flu, or allergies?" Your HVAC fault detection is classification: "Is this sensor reading normal, or is there a fault? And if there's a fault, what type?"

Binary vs Multi-Class

Binary: Two classes. Fault / No Fault. Spam / Not Spam. This is your primary use case.

Multi-class: Multiple fault types. Could be: "normal," "stuck valve," "sensor drift," "refrigerant leak," etc.

The Classification Workflow (Your HVAC Fault Detection)

import pandas as pd
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split
from sklearn.metrics import classification_report
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline

# 1. Load your sensor data with engineered features
df = pd.read_csv('hvac_features.csv')

# 2. Define features (X) and label (y)
feature_cols = [
    'temp_rolling_mean_24h', 'temp_rolling_std_24h',
    'delta_t', 'setpoint_deviation',
    'hour', 'is_business_hours',
    'temp_rate_of_change', 'runtime_hours'
]
X = df[feature_cols]
y = df['is_fault']  # 0 = normal, 1 = fault

# 3. Time-based split (NOT random for time-series!)
split_idx = int(len(df) * 0.8)
X_train, X_test = X[:split_idx], X[split_idx:]
y_train, y_test = y[:split_idx], y[split_idx:]

# 4. Build and train pipeline
pipeline = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', RandomForestClassifier(
        n_estimators=100, max_depth=10,
        class_weight='balanced'  # handles imbalanced data!
    ))
])
pipeline.fit(X_train, y_train)

# 5. Evaluate
y_pred = pipeline.predict(X_test)
print(classification_report(y_test, y_pred))
INTERVIEW TIP

Notice class_weight='balanced' above. This is critical for imbalanced data. In HVAC, most readings are normal (maybe 95%) and faults are rare (5%). Without class weighting, the model will just predict "normal" every time and get 95% accuracy while being completely useless. class_weight='balanced' tells the model to pay more attention to the rare fault class. Always mention this when discussing fault detection.

Time-Series Basics

ANALOGY

Time-series data is like a heartbeat monitor. The order matters. You can't shuffle the readings and expect them to make sense. Each point depends on what came before it. This is fundamentally different from tabular data like "predict house prices from square footage and bedrooms," where row order doesn't matter.

Why Time-Series Is Special

  • Temporal dependencies: Today's reading is correlated with yesterday's. You can't treat rows as independent.
  • Seasonality: HVAC data has daily patterns (day vs night), weekly patterns (weekday vs weekend), and annual patterns (summer vs winter).
  • Trend: Equipment degrades over time. A slow upward trend in energy consumption might signal declining efficiency.
  • No random splits: As discussed, you MUST split chronologically.

Time-Series Cross-Validation

Standard k-fold won't work. Use TimeSeriesSplit:

from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5)

# Each fold trains on all prior data, tests on next chunk:
# Fold 1: Train [---]  Test [--]
# Fold 2: Train [-----]  Test [--]
# Fold 3: Train [-------]  Test [--]
# Fold 4: Train [---------]  Test [--]
# Fold 5: Train [-----------]  Test [--]
YOUR JOULEA CONTEXT

"We processed 10M+ data points daily from building sensors, which is inherently time-series data. We engineered temporal features like hour-of-day, rolling statistics, and lag features to capture the patterns in HVAC behavior. For validation, we used TimeSeriesSplit rather than random cross-validation to avoid temporal leakage. Seasonal patterns were important because HVAC fault signatures look different in summer cooling mode versus winter heating mode."

Embeddings & Semantic Similarity

ANALOGY

Imagine you're organizing books in a library. Traditional keyword search is like organizing alphabetically: "HVAC troubleshooting" and "air conditioning repair guide" end up in completely different sections even though they're about the same thing. Embeddings are like organizing by meaning: both books end up in the same neighborhood because they cover the same topic, even though they use different words. An embedding converts text into a list of numbers (a vector) that captures its meaning.

What Is an Embedding?

An embedding is a numerical representation (vector) of text that captures semantic meaning. Similar concepts end up with similar vectors, measured by distance or angle between them.

The sentence "The AC unit is broken" might become a vector like [0.23, -0.45, 0.89, ..., 0.12] (typically 768 to 1536 dimensions). The sentence "The air conditioning system has a malfunction" would produce a very similar vector, even though the words are completely different.

How Similarity Is Measured

MethodHow It WorksRangeWhen to Use
Cosine Similarity Measures the angle between two vectors -1 to 1 (1 = identical) Most common for text embeddings. What FAISS uses by default.
Euclidean Distance (L2) Straight-line distance between vectors 0 to infinity (0 = identical) Works when vector magnitudes are normalized
Dot Product Sum of element-wise products -inf to inf Fast computation, used when vectors are normalized (equivalent to cosine)

Embedding Models

Different models produce embeddings of different quality and dimensions:

  • OpenAI text-embedding-ada-002: 1536 dimensions. Good general-purpose. What you'd use via the OpenAI API.
  • OpenAI text-embedding-3-small/large: Newer, better performance, configurable dimensions.
  • Sentence-BERT (all-MiniLM-L6-v2): Open source, 384 dimensions, runs locally. Good for cost-sensitive applications.
  • Cohere embed-v3: Another commercial option with strong performance.

In Code

from openai import OpenAI
import numpy as np

client = OpenAI()

# Generate an embedding
response = client.embeddings.create(
    input="HVAC supply air temperature sensor fault",
    model="text-embedding-ada-002"
)
vector = response.data[0].embedding  # list of 1536 floats

# Cosine similarity between two vectors
def cosine_similarity(a, b):
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# similarity = 0.92 means very semantically similar

This is what made the chatbot able to answer questions about specific buildings even though the LLM had never been trained on our data."

Vector Databases (FAISS & Pinecone)

ANALOGY

A vector database is like a smart librarian. Instead of looking up books by exact title (keyword search), you describe what you're looking for in plain language, and the librarian finds the most relevant books based on meaning. Under the hood, it's organizing millions of vectors (book representations) so that finding the nearest neighbors is fast, even at scale. Without a vector DB, you'd have to compare your query against every single document, which is impossibly slow.

FAISS (Facebook AI Similarity Search)

What it is: An open-source library (not a database service) for fast similarity search over dense vectors. Developed by Meta. Runs locally in-memory.

When to use it: Smaller to medium datasets (thousands to low millions of vectors), or when you don't want to pay for a managed service. Great for prototyping and when your data fits in memory.

import faiss
import numpy as np

# Create an index for 1536-dimensional vectors
dimension = 1536
index = faiss.IndexFlatL2(dimension)  # exact L2 search

# Add document vectors (numpy array, shape: [n_docs, dimension])
doc_vectors = np.array(all_embeddings).astype('float32')
index.add(doc_vectors)

# Search: find 5 nearest neighbors to a query
query_vector = np.array([query_embedding]).astype('float32')
distances, indices = index.search(query_vector, k=5)

# indices[0] gives you the IDs of the 5 most similar documents

Pinecone

What it is: A managed, cloud-hosted vector database service. No infrastructure to manage.

When to use it: Production systems at scale, when you need persistence, filtering (metadata), and don't want to manage infrastructure. Costs money but saves engineering time.

FAISS gave us sub-millisecond retrieval for our vector searches. If we needed to scale to hundreds of thousands of documents or add complex metadata filtering, like filtering by building ID or document type, Pinecone would have been the better choice."

RAG Architecture

ANALOGY

RAG is like an open-book exam. The student (LLM) is smart and knows how to reason, but instead of relying purely on memory, they get to look up specific pages in their textbook before answering each question. The retrieval step finds the relevant pages; the generation step writes the answer using those pages as context. Without RAG, the LLM would hallucinate answers from memory. With RAG, it grounds its answers in your actual documents.

The RAG Pipeline

User Query
→
Embed Query
→
Vector Search
→
Retrieve Docs
→
LLM + Context
→
Answer

Step by Step

1. Ingestion (offline, done once)

  • Take your documents (PDFs, manuals, specs)
  • Chunk them into smaller pieces (e.g., 500-1000 tokens each, with overlap)
  • Embed each chunk into a vector
  • Store vectors + metadata in a vector database (FAISS, Pinecone)

2. Retrieval (at query time)

  • User asks a question
  • Embed the question using the same embedding model
  • Search the vector DB for the k most similar chunks (typically k = 3-5)

3. Generation (at query time)

  • Construct a prompt: system instructions + retrieved chunks + user question
  • Send to LLM
  • LLM generates answer grounded in the retrieved context

Chunking Strategy

How you split documents matters a lot:

StrategyHowTrade-off
Fixed-size Split every N tokens Simple but may split mid-sentence
Recursive character Split on paragraphs, then sentences, then characters Better semantic boundaries. LangChain default.
Semantic Use embeddings to detect topic shifts Best quality but slower and more complex

Overlap: Chunks should overlap (e.g., 100-200 tokens) so context isn't lost at boundaries. If chunk 1 ends mid-explanation, chunk 2 repeats that section so it's retrievable from either.

Key Design Decisions in RAG

DECISIONS YOU SHOULD BE ABLE TO DISCUSS
  • Chunk size: Smaller chunks = more precise retrieval but might miss context. Larger chunks = more context but might dilute relevance. 500-1000 tokens is typical.
  • Number of retrieved chunks (k): Too few = might miss relevant info. Too many = fills up the LLM context window with noise. 3-5 is common.
  • Embedding model choice: Trade-off between quality and cost/speed.
  • Reranking: After initial vector search, use a cross-encoder to rerank results for better precision. Adds latency but improves quality.
  • Conversation memory: For multi-turn chats, you need to either include conversation history in the prompt or summarize it.
YOUR JOULEA CONTEXT

"We built an agent-based RAG system for building management support. Building documentation, equipment specs, and troubleshooting guides were chunked using recursive character splitting, embedded, and stored in FAISS. At query time, we retrieved the top relevant chunks and provided them as context to the LLM via LangChain. We implemented conversation memory with context windowing so multi-turn conversations maintained coherence without exceeding the context window. The system achieved 85% accuracy validated by subject matter experts."

LangChain & LangGraph

ANALOGY

LangChain is to LLM applications what Express.js is to web servers. Express doesn't make HTTP requests for you, but it gives you middleware, routing, and structure so you can build web apps faster. LangChain doesn't run the LLM, but it gives you chains, prompts, retrievers, memory, and tooling so you can build LLM-powered applications faster instead of duct-taping raw API calls together.

LangGraph is the upgrade: it's like going from Express middleware (linear pipeline) to a state machine where your AI agent can loop, branch, and make decisions about what to do next.

LangChain Core Concepts

Chains

A chain is a sequence of steps. The simplest: take user input, format a prompt, send to LLM, return output.

from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

# Define a prompt template
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an HVAC expert. Answer using the context: {context}"),
    ("human", "{question}")
])

# Create the chain: prompt -> LLM -> output
llm = ChatOpenAI(model="gpt-4")
chain = prompt | llm  # LCEL (LangChain Expression Language) pipe syntax

# Run it
response = chain.invoke({
    "context": retrieved_docs,
    "question": "Why is Building 42's AHU running constantly?"
})

Retrievers

A retriever wraps your vector DB and provides a standard interface for fetching relevant documents:

from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

# Load existing FAISS index
vectorstore = FAISS.load_local("faiss_index", OpenAIEmbeddings())

# Create a retriever (returns top 4 similar docs)
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

Memory

Memory lets the chain remember previous messages in a conversation:

from langchain.memory import ConversationBufferWindowMemory

# Keep last 10 exchanges (window memory)
memory = ConversationBufferWindowMemory(k=10, return_messages=True)

# Types of memory:
# ConversationBufferMemory     - stores everything (expensive for long chats)
# ConversationBufferWindowMemory - last k messages (your choice)
# ConversationSummaryMemory    - LLM summarizes old messages (saves tokens)

LangGraph: Agent State Machines

LangGraph lets you build agents that can decide what to do next based on the current state. Instead of a linear chain, you define a graph of nodes (actions) and edges (transitions).

from langgraph.graph import StateGraph, END

# Define the state
class AgentState(TypedDict):
    messages: list
    context: str
    should_retrieve: bool

# Define nodes (functions that process state)
def classify_query(state):
    # Decide if we need to retrieve documents or can answer directly
    ...

def retrieve_docs(state):
    # Fetch relevant documents from vector DB
    ...

def generate_response(state):
    # Generate answer using LLM + context
    ...

# Build the graph
graph = StateGraph(AgentState)
graph.add_node("classify", classify_query)
graph.add_node("retrieve", retrieve_docs)
graph.add_node("generate", generate_response)

# Define edges (routing logic)
graph.add_edge("classify", "retrieve")  # or skip to generate
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)

app = graph.compile()
LANGCHAIN VS LANGGRAPH

LangChain: Linear pipelines. Input goes through a sequence of steps. Great for straightforward RAG, simple chains.
LangGraph: Cyclic graphs with branching logic. The agent can loop back, try different tools, or take different paths based on the query. Use it when your agent needs to make decisions, like "should I search the docs, call an API, or ask for clarification?"

LlamaIndex

ANALOGY

If LangChain is a general-purpose LLM framework (like a Swiss army knife), LlamaIndex is a specialized tool built specifically for the data ingestion and retrieval part of the pipeline. Think of LlamaIndex as the librarian and LangChain as the entire library system (librarian + checkout desk + event space + cafe). LlamaIndex is laser-focused on connecting your data to LLMs.

Core Abstractions

Documents & Nodes

A Document is a container for your raw data (a PDF, a web page). LlamaIndex splits Documents into Nodes (chunks) with metadata and relationships preserved.

Indexes

An Index structures your Nodes for retrieval. The most common is VectorStoreIndex, which embeds nodes and enables similarity search.

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

# Load all documents from a directory
documents = SimpleDirectoryReader("./building_docs").load_data()

# Build the index (embeds all chunks automatically)
index = VectorStoreIndex.from_documents(documents)

# Query it
query_engine = index.as_query_engine()
response = query_engine.query(
    "What is the recommended maintenance schedule for AHU-3?"
)
print(response)

Prompt Engineering

ANALOGY

Prompt engineering is like being a really good manager giving instructions to a highly capable but literal new employee. The employee (LLM) can do amazing work, but the quality of their output depends entirely on how clearly and specifically you tell them what you want, what format to use, what constraints to follow, and what examples to reference. Vague instructions get vague results.

Key Techniques

Few-Shot Prompting

Provide examples of the input/output format you want. This was critical for your data labeling workflow:

# Your data labeling prompt might look like:
"""
You are a sensor data normalization expert. Given a raw sensor label,
output the standardized label.

Examples:
- Input: "TMP_SUPPLY" -> Output: "supply_air_temperature"
- Input: "RH-ZONE1"   -> Output: "zone_1_relative_humidity"
- Input: "AHU3_SAT"   -> Output: "ahu_3_supply_air_temperature"

Now normalize this label:
Input: "{raw_label}"
"""

Chain-of-Thought (CoT)

Ask the model to reason step by step. Improves accuracy on complex problems:

"Think through this step by step before giving your answer."

System Prompts

Set the model's role, constraints, and behavior upfront. For the chatbot:

"You are a building management support assistant. Only answer questions
using the provided context documents. If the answer is not in the
context, say 'I don't have enough information to answer that.'
Never make up technical specifications."

Output Formatting

Tell the model exactly what format you want:

"Respond in JSON format with keys: label, confidence, reasoning"

For the chatbot, we used detailed system prompts to constrain the model to only answer from retrieved context, reducing hallucination. This combination of prompt engineering with RAG retrieval is what got us to 85% accuracy."

Manual mapping. An engineer looks at each raw label, figures out what it means, and maps it to the standard schema. Slow, error-prone, doesn't scale as you onboard new buildings.

Your Solution

An OpenAI API-based pipeline using few-shot prompting. You provided examples of known mappings, and the model generalized to handle new labels it hadn't seen before. This is fundamentally an NLP classification task using in-context learning rather than traditional model training.

Even though you didn't train a model from scratch, this is a legitimate machine learning approach:

We used F1 score as our primary metric since it balances both, but we leaned toward higher recall."

Domain-specific features like delta-T, the difference between supply and return air temperatures, which is a core HVAC diagnostic metric, and setpoint deviation, how far actual conditions are from the target.
