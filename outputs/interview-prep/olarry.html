01 How would you design a document ingestion pipeline that classifies and extracts structured data from tax forms (W-2s, K-1s, 1099s)? ▼

This is their core product challenge. Walk through:

  • Ingestion: Upload to S3, trigger Lambda/async worker. Support PDF, images, scanned docs.
  • Classification: LLM or fine-tuned model to identify form type. Similar to your IoT data labeling workflow.
  • Extraction: OCR (Textract) + LLM for structured field extraction. Validate against known schemas for each form type.
  • Validation: Human-in-the-loop for low-confidence extractions. CPA reviews flagged items.
  • Storage: Structured output to PostgreSQL, raw docs in S3. Audit trail for compliance.

Tie back to your Joulea experience: "I built a similar pipeline for IoT sensor data. The pattern is the same: unstructured input, classification, extraction, validation."

02 We're acquiring CPA firms. How would you approach migrating their client data into our platform? ▼
  • Audit: Map their data formats, schemas, and systems. Every firm will be different.
  • ETL Pipeline: Build configurable ingestion (like AWS Glue jobs) that can adapt to different source formats
  • Validation: Checksums, record counts, client-visible verification step
  • Incremental rollout: Migrate one firm at a time, validate, then template the process
  • Reference your T-Mobile SOAP→REST work and S3/Glue migration at Joulea
03 How do you approach building a RAG system? What are the failure modes? ▼

You've built one. Speak from experience:

  • Chunking strategy matters: Tax documents have structured sections. Chunk by logical units (form fields, sections), not arbitrary token counts.
  • Embedding model choice: Balance cost vs. accuracy. Start with OpenAI embeddings, benchmark against open-source.
  • Failure modes: Hallucination on edge cases, retrieval returning irrelevant chunks, stale index after document updates, context window overflow on complex queries.
  • Mitigations: Citation/source tracking (you did this at Joulea), confidence scoring, human-in-the-loop for tax-sensitive outputs.
04 Walk me through how you'd build a React component that handles complex form state (multi-step tax questionnaire). ▼
  • State management: useReducer or Redux for complex, multi-step state. You used Redux at Joulea.
  • Validation: Schema-based (Zod/Yup) with per-step validation before advancing
  • Persistence: Auto-save to API on each step completion so users don't lose progress
  • Conditional logic: Tax forms have branching paths (business vs. individual, state-specific). Drive UI from a config/schema rather than hardcoding branches.
  • Accessibility: Progress indicator, keyboard nav, error summaries
05 How do you handle data security and compliance in a system that processes sensitive financial data? ▼

You've done PCI compliance at T-Mobile. Lean on that:

  • Encryption: At rest (AES-256, KMS) and in transit (TLS). Joulea does "proactive data monitoring in transit and at rest."
  • Access control: RBAC, principle of least privilege. OAuth 2.0 (you implemented this at T-Mobile).
  • Audit logging: Every data access logged. You built Splunk dashboards at T-Mobile for monitoring.
  • Data isolation: Multi-tenant architecture with strict tenant boundaries. Tax data can't leak between clients.
  • SOC 2 / IRS safeguards: Show awareness that tax data has specific regulatory requirements beyond general web app security.
